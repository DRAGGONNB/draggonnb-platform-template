The product in 60 seconds

The DraggonnB OS thesis

DraggonnB is not "a CRM with a hospitality module." It is a three-layer ecosystem for South African operators in tourism, hunting, hospitality, restaurants, and community safety:

(1) Operational Core — multi-tenant database of record with RLS-isolated tenants, modular activation, automated provisioning, and 10 production AI agents. (2) AI Agency & Approval Spine — per-role agents that propose, owners that approve, RLS enforcing who can do what. (3) Public Network Surfaces — vertical sites (GoHunting.com) and location marketplaces (Plett-stays) that turn operational data into demand-generation.

Anchored in the Southern African hunting + game-lodge vertical, the same pattern reuses across every adjacent vertical (guest houses, fishing, safari, restaurants, residential security). Pilot lodge: Swazulu Game Lodge (uses accommodation + trophy modules; their domain swazulu.com linked once stable).

The three-layer ecosystem

What you're building, in one picture. Layer 1 is shipped. Layer 2 is the next architectural shift (Wave A). Layer 3 is the investor-narrative play (Wave C, but the pitch artifact lives in Wave A).

Layer 3 — Public network surfaces

Vertical sites (GoHunting.com) and location marketplaces (plett-stays.co.za-style) that aggregate operational data into discoverable, monetisable destinations. Vendors subscribe for placement; operators feed approved content; guests generate traffic. Network effects: every new operator makes the public surface richer, which generates leads back to operators.

GoHunting.com (vertical) Plett guest-house marketplace (location) Go-X template (reusable) Vendor subscriptions = revenue layer

Layer 2 — AI agency & approval spine

Per-role agents (owner / manager / lodge / vendor) that propose actions; humans approve via Telegram or web. RLS enforces who can approve what. Damage auto-billing, rate adjustments, special offers, content publishing, vendor jobs — all flow through the same approval queue. Turns AI from "generator" into "agent" without losing human control.

approval_requests table (new) Per-role permission lattice Owner Telegram tap-to-approve Damage flag → auto-charge

Layer 1 — Operational core (shipped today)

Multi-tenant Next.js + Supabase platform with RLS-isolated tenants. 11 modules (CRM, email, social, content studio, accommodation, restaurant, elijah, autopilot, AI agents, analytics, billing) gated by tenant_modules. 9-step automated provisioning saga delivers a working tenant in 3 business days. 10 production AI agents extending BaseAgent with cost ceilings, advisory locks, and per-tenant cache isolation.

Three-channel communication doctrine

Architectural test for any new comms flow: is this staff-facing, client-facing, or experience-facing? Mismatched routing burns money on per-message API costs.

WhatsApp Cloud API

What "things nobody else has" means concretely

Five differentiators from the strategic backlog. Three of them ship in Wave A (60 days).

▶ Wave A · 60 days

Owner-approved AI agent loop

AI drafts a special / rate change / damage charge → owner taps approve in Telegram → system executes (post to social, update PMS, charge guest). Existing PMS systems (Cloudbeds, Little Hotelier, RoomRaccoon) have no AI agent layer at all.

▶ Wave A · 60 days

Damage auto-billing in seconds

Staff Telegram flag → system auto-calculates → guest WhatsApp charge confirmation. End-to-end in seconds. Closest competitor stack: SiteMinder + Stripe + manual reconciliation = ~30 minutes per incident.

▶ Wave A + B · 90 days

Carcass-routing pipeline with vendor SOPs

Lodge → butcher → taxidermist with deposits, special instructions, notifications, approval gates. No global hunting software has this. SA hunting tools today = spreadsheet replacements.

▶ Wave B · 120 days

Genetic-tree game-farm tracking

Farmer Telegram photo upload → AI auto-classifies (species, body condition, horn measurements) → logs to hierarchical genetic tree. Replaces 3 competing apps farmers currently use badly.

▶ Wave C · 120 days+

Closed-loop content marketplace

Per-stay PWA → guest content upload → approval → publish to Layer 3 public site → drives leads back to the lodge. Network effect: every guest stay enriches the public surface, which feeds the next booking.

● Already shipped

11-module multi-tenant SA-localised platform

ZAR-native (PayFast + 15% VAT), 3-business-day automated provisioning, RLS-isolated tenants, brand-voice AI agents, wildcard subdomains. No competing platform combines this scope with SA-specific operational depth.
