  <action>
1. Read `lib/supabase/server.ts` and `lib/supabase/middleware.ts` first to understand the current `get/set/remove` callback shape and any wrapped logic (tenant resolution, header injection, etc.).

2. Bump dependencies in `package.json`:
   - `"@supabase/ssr"`: `"^0.1.0"` -> `"^0.10.2"`
   - `"@supabase/supabase-js"`: `"^2.39.0"` -> `"^2.105.1"`
   - Add `"jose"`: `"^5.10.0"` to dependencies (NOT devDependencies — middleware imports it at runtime).

   Run `npm install` to refresh `package-lock.json`. Confirm no peer-dep warnings or audit-critical CVEs (npm warn is acceptable; npm err is not).

3. Refactor `lib/supabase/server.ts`:
   - Replace the `cookies: { get, set, remove }` callback object inside `createServerClient(...)` with `cookies: { getAll, setAll }`.
   - `getAll()` returns `cookieStore.getAll()` (the Next.js cookies() result).
   - `setAll(cookiesToSet)` iterates `cookiesToSet.forEach(({ name, value, options }) => cookieStore.set(name, value, options))` wrapped in try/catch — calling from a Server Component throws but is safe to ignore (Supabase pattern).
   - Preserve the `await cookies()` async call at the top (Next.js 14 pattern).
   - Keep the function signature `createClient()` unchanged (callers must not change).
   - Reference exact pattern from 13-RESEARCH.md Section 8 "New `lib/supabase/server.ts` after upgrade".

4. Refactor `lib/supabase/middleware.ts` `updateSession()`:
   - Replace `cookies: { get, set, remove }` with `cookies: { getAll, setAll }`.
   - `getAll()` returns `request.cookies.getAll()`.
   - `setAll(cookiesToSet)` iterates twice — once writing to `request.cookies.set(name, value)` (so subsequent code in the same request sees the cookie), then re-creates `response = NextResponse.next({ request })`, then iterates again writing to `response.cookies.set(name, value, options)`.
   - **CRITICAL — CATASTROPHIC #1 guard:** Verify the `setAll` callback NEVER spreads or includes a `domain` field in its options. The Supabase library should NOT inject `Domain=.draggonnb.co.za`. Add a defensive comment block above setAll:
     ```typescript
     // SSO-07 / CATASTROPHIC #1: per-host cookies only.
     // NEVER pass Domain=.draggonnb.co.za — it would leak the session cookie
     // across tenant subdomains and trophyos.co.za. Each host owns its own cookie.
     // If you need to add cookie options here, add a runtime assertion that options.domain is undefined.
     ```
   - Preserve all surrounding logic: the existing `resolveTenant()` call, `x-tenant-*` header injection, protected-route checks, and any tenant cache. Do NOT rewrite tenant resolution — only the cookie callback section.
   - Preserve the function signature and return type — `updateSession(request: NextRequest)` returns `NextResponse`.

5. Run `npx tsc --noEmit` to confirm zero type errors. The 0.10.2 cookie API exposes `CookieOptions` differently — if a type error surfaces in `setAll`, type the parameter as `{ name: string; value: string; options: CookieOptions }[]` (import `CookieOptions` from `@supabase/ssr`).

6. Run the existing test suite: `npx vitest run` (full suite). Expected: all 720+ tests pass. If middleware integration tests fail, the `setAll` two-step (request.cookies.set then response.cookies.set) is the most likely culprit — re-check against the 13-RESEARCH.md pattern.

WHY refactor in same plan as version bump: 13-RESEARCH.md LATENT-01 confirms TypeScript will NOT catch the API mismatch (old shape still type-checks in 0.1.0; in 0.10.2 the missing methods silently no-op rather than throwing). Splitting npm install + refactor across plans risks shipping a broken auth state between waves.

WHY include jose now: Plan 13-06 imports it via `@draggonnb/federation-shared`, but the package itself (Plan 13-05) needs jose available locally during DraggonnB development for type-checking. Adding it here to the consumer means no install gap downstream.
