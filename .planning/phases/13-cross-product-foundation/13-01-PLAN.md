2. Step 1 (capture token): Build a PayFast Subscribe form payload (subscription_type=2 ad-hoc, amount=5.00 rands, notify_url=user-provided HTTPS URL, return_url, cancel_url, item_name=`SPIKE-13-01`, m_payment_id=`SPIKE-{timestamp}`). Compute the MD5 signature per the existing pattern in `lib/payments/payfast.ts` (read that file to mirror exactly: alphabetical sort, URL-encode, append `&passphrase={pp}` if set). PRINT the form-encoded redirect URL to stdout — operator manually completes the Subscribe checkout in browser, captures the `token` returned in the ITN, then re-runs the script with `PAYFAST_TEST_TOKEN={token}` env var.

3. Step 2 (amount-unit test): With `PAYFAST_TEST_TOKEN` set, POST to `https://sandbox.payfast.co.za/eng/recurring/ad-hoc/{token}` (or whatever exact endpoint is in `lib/payments/payfast-subscription-api.ts` — read that file and reuse the URL+signature pattern). Run TWO calls:
   - Call A: `amount=250.00` (rands as string, 2 decimals)
   - Call B: `amount=25000` (cents as integer/string, no decimals)
   Capture both responses (status, headers, body) and write to `scripts/spikes/payfast-spike-output.json`. The call that returns success (HTTP 2xx + non-error body) tells us the unit.

4. Step 3 (arbitrary-amount + idempotency test): Issue a second adhoc charge against the SAME captured token at a different amount (e.g., `amount=75.00` if rands won, else `7500`). Confirm success — proves the token accepts arbitrary amounts (not locked to one value). Then re-issue with the same `m_payment_id` to verify PayFast returns a duplicate-detection response (idempotency confirms).

5. Step 4 (zero/floor test): POST `amount=0` (or `0.00`). Expected: error response. Capture the error body to confirm the floor constraint exists.

6. Step 5 (hold-and-capture probe): Inspect both successful response bodies for `capture_url`, `hold_reference`, `auth_code`, or any field suggesting authorize-now-capture-later. Absence + checking the latest `payfast-subscription-api.ts` route documentation (https://developers.payfast.co.za/docs#recurring_billing) confirms hold-and-capture is unavailable.

WHY a one-off script and not a vitest test: vitest mocks PayFast in unit tests (correct for production code). The spike needs LIVE network calls against `sandbox.payfast.co.za`. Keep this script outside `__tests__` so CI does not run it.

WHY mirror existing signature/URL patterns from `payfast.ts` and `payfast-subscription-api.ts`: the spike must validate the SAME signature method our production code uses; otherwise we prove nothing.

Do NOT commit `scripts/spikes/payfast-spike-output.json` (raw responses may include sensitive headers). Add to `.gitignore` if not already.
