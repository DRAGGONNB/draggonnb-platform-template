# Phase 08.2: Multi-Tenant WhatsApp Client Refactor

## Goal
Refactor the WhatsApp client to resolve WABA credentials per-tenant from `tenant_modules.config` instead of relying on global environment variables. Backward-compatible: falls back to env vars when no tenant config exists.

## Prerequisites
- [ ] Phase 08.1 complete (tenant config shape defined and stored)
- [ ] At least one org has WABA credentials in `tenant_modules.config`

## Tasks

### Task 1: Tenant WhatsApp Config Resolver
**File:** `lib/meta/whatsapp-tenant.ts`
**Action:** Create
**Details:**
- `getTenantWhatsAppConfig(orgId: string): Promise<WhatsAppTenantConfig | null>`
- Queries `tenant_modules` WHERE `module_id = 'whatsapp'` (or from `config` JSONB on social/accommodation module)
- Returns: `{ accessToken, phoneNumberId, wabaId, onboardingModel }`
- Falls back to global env vars if no tenant config found
- Per-request cache using module-level Map (cleared on request boundary)
- Type: `WhatsAppTenantConfig` interface

### Task 2: Refactor WhatsApp Client
**File:** `lib/whatsapp/client.ts`
**Action:** Modify
**Details:**
- Add optional `orgId?: string` parameter to all public functions:
  - `sendTextMessage(to, text, orgId?)`
  - `sendTemplateMessage(to, template, orgId?)`
  - `sendMediaMessage(to, media, orgId?)`
  - `sendInteractiveMessage(to, interactive, orgId?)`
- Internal `getCredentials(orgId?)` helper:
  - If orgId provided: call `getTenantWhatsAppConfig(orgId)`
  - If not or null result: use `process.env.WHATSAPP_ACCESS_TOKEN` / `WHATSAPP_PHONE_NUMBER_ID`
- All Graph API calls use resolved credentials
- Zero breaking changes to existing callers (orgId is optional)

### Task 3: Refactor WhatsApp Router
**File:** `lib/whatsapp/router.ts`
**Action:** Modify
**Details:**
- After resolving org from inbound phone number lookup, pass `orgId` to response functions
- `routeMessage(message, metadata)` now extracts `phone_number_id` from metadata
- Uses `phone_number_id` to map to org (reverse lookup via `tenant_modules.config`)
- Passes resolved orgId to `sendTextMessage()` etc.

### Task 4: Multi-WABA Webhook Handler
**File:** `app/api/webhooks/whatsapp/route.ts`
**Action:** Modify
**Details:**
- Inbound webhooks from Meta include `metadata.phone_number_id`
- Create lookup: `resolveOrgFromPhoneNumberId(phoneNumberId: string)`
  - Query `tenant_modules` where `config->>'phone_number_id' = phoneNumberId`
  - Cache result (phone_number_id -> org_id mapping rarely changes)
- Pass resolved orgId to `routeMessage()`
- HMAC validation: may need per-tenant app secret (or use global Meta App secret)
- Support multiple WABAs receiving messages to same webhook endpoint

### Task 5: Update N8N WhatsApp Workflow Templates
**Files:** `n8n/wf-whatsapp-booking-confirm.json`, `n8n/wf-whatsapp-reminder.json`
**Action:** Modify
**Details:**
- HTTP Request nodes: add `org_id` header or query param to API calls
- API routes (`/api/accommodation/...`) resolve tenant WABA from org_id in request
- N8N can use `$env.DEFAULT_ORG_ID` as fallback

## Verification
- [ ] `sendTextMessage("...", "test", "org-id-1")` uses org-1's WABA credentials
- [ ] `sendTextMessage("...", "test")` still uses global env vars (backward compat)
- [ ] Inbound webhook correctly routes message to org based on phone_number_id
- [ ] WhatsApp router responses use correct org's credentials
- [ ] N8N workflows pass org_id to API endpoints
